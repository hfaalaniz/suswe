ETAPA 2 — Bugs en el firmware ESP32 descubiertos al leer el .ino
Archivos afectados: firmware/SU900_ESP32_Gateway/SU900_ESP32_Gateway.ino
Por qué antes que OTA/AP: al revisar el firmware aparecen 4 bugs que afectan la operación básica actual, conviene resolverlos antes de agregar funcionalidades encima.

Bugs encontrados en el .ino:

Registros de monitoreo incorrectos — handleMonitor() lee 16 registros desde REG_STATUS = 0x3000, pero según el manual los registros U0 están en 0x1001~0x1020. El firmware usa un mapa de nombres y escalas hardcodeado que no coincide con los del backend Node.js ni con MONITOR_REGISTERS del frontend.

handleStatus() mezcla registros de estado con U0 — lee 0x3000 (estado) + 3 registros contiguos (0x3001~0x3003) asumiendo que son frecuencia/corriente/tensión de salida. Pero 0x3001~0x3003 son parámetros P3 (boost de par, frecuencia boost, voltaje V/F), no variables de monitoreo. El monitoreo real está en 0x1001~0x1004.

Valores de comandos incorrectos — CMD_RAMP_STOP = 0x0005, CMD_FREE_STOP = 0x0006 están invertidos respecto al manual (que dice: 5=paro libre, 6=desaceleración). Además CMD_JOG_FORWARD = 0x0007 y CMD_FAULT_RESET = 0x0409 no coinciden con el manual (JOG_FWD=3, JOG_REV=4, FREE_STOP=5, RAMP_STOP=6, FAULT_RESET=7).

handleFaultHistory() usa direcciones incorrectas — lee desde REG_FAULT_H1 = 0xA000, pero el manual indica que el historial de fallas está en P9-14=0x900E, P9-15=0x900F, P9-16=0x9010.

rawToReal() en el firmware es incompleta — solo tiene escalas para ~10 parámetros, ignora todo P2, P3, P4, P5, P6, Pd. Los valores escritos y leídos de esos grupos llegan sin conversión.

handleDigitalOutputs() recibe bits como entero en vez de objeto — el backend Node.js y el frontend envían { bits: { DO1: bool, DO2: bool, ... } }, pero el firmware espera { bits: N } (entero directo).

Auth endpoints sin prefijo /api/ — el firmware los registra como /auth-status y /auth-config, pero el frontend llama a /api/auth-status y /api/auth-config.
